# Optic — UX/UI review · 2026-05-29 · Phase 1.0-A Foundation

**Scope:** `apps/admin/app/**`, `packages/ui/src/**` (primitives).

---

## Findings

### F-Optic-01 · Polish · Affordance: cards are full-area links but visual hover is subtle

`apps/admin/app/(authed)/page.tsx:120-132` — the dashboard cards render as `<a className="adm-card">` with `text-decoration:none` inline. The `.adm-card` class in `apps/admin/app/globals.css:234-242` has NO `:hover` rule, so users get only the default link-cursor and the unstyled focus ring (from `a:focus-visible` global rule). Click affordance is thin for what is the cpanel's primary navigation surface.

Fix: add `.adm-card:hover { border-color: var(--brand-orange); background: var(--surface); }` paired with `.adm-card:focus-visible` ring tied to `--focus-ring`. Mirror the `az-card--interactive` pattern from `packages/ui/src/Card/Card.css:48-62`.

### F-Optic-02 · Major · Single-h1 per page is correct, but landmark hierarchy missing on landing pages

`apps/admin/app/page.tsx` and `apps/admin/app/sign-in/page.tsx` render `<main id="main">` directly. The unauthenticated landing has no `<nav>` or `<header role="banner">` landmarks beyond the inline header div. Authenticated cpanel layout in `(authed)/layout.tsx` does it correctly (header + nav + main). Outside the (authed) group, screen-reader users navigating by landmark get only "main" — fine for thin pages but worth a consistent header landmark.

Fix: wrap landing's inline `<header style={...}>` in a semantic `<header>` with `role="banner"` or simply a `<header>` tag. Same for `sign-in/page.tsx` line 28.

### F-Optic-03 · Major · `not-found.tsx` renders its OWN `<main id="main">` which can duplicate the (authed) layout's landmark

`apps/admin/app/not-found.tsx:6` — when an unmatched route falls inside the `(authed)` segment, Next.js may resolve the segment's `layout.tsx` (which provides `<main id="main">` at `(authed)/layout.tsx:85`) AND render `not-found.tsx` inside it (which provides its own `<main id="main">`). That gives two `<main>` elements with the same id — SC 4.1.1 / duplicate-landmark warning.

Fix: define a segment-scoped `(authed)/not-found.tsx` that returns a `<section>` (not `<main>`), keeping the root `not-found.tsx` for unmatched routes outside the segment. Or remove the root `<main>` from `not-found.tsx` and instead structure it as a `<section>` rendered inside the root layout's body.

### F-Optic-04 · Minor · Skip-link target exists but landing-page `<main id="main">` lacks `tabIndex={-1}`

`apps/admin/app/page.tsx:12` and the (authed) layout main do not set `tabIndex={-1}`. Per SC 2.4.1 best practice, the skip-link target must be programmatically focusable so the browser actually moves focus on activation. Without `tabIndex={-1}` Safari/Firefox sometimes leave focus on the link.

Fix: `<main id="main" tabIndex={-1}>` everywhere `<main>` appears.

### F-Optic-05 · Minor · Inline styles on every page should consolidate into utility classes (Hick's law / consistency)

Every authed page repeats the same eyebrow/kicker styling inline (e.g. `tenants/page.tsx:8-16`, `costs/page.tsx:8-16`, `activity/page.tsx:8-16`). Future Halo/Canon audits will be cleaner if this kicker pattern is moved to `.adm-kicker` and the headings to `.adm-page-h1`. Currently the inline pattern is duplicated ~12 times.

Fix: extract `.adm-kicker` + `.adm-page-h1` + `.adm-page-body` rules in `globals.css` and apply via className. Pure DX/consistency improvement.

### F-Optic-06 · Polish · Authed dashboard has 11 cards in a single auto-fit grid — wayfinding could chunk them by purpose

`apps/admin/app/(authed)/page.tsx:14-81` lists 11 sections side-by-side. PRD §3.2 chunks them as runtime / financial / compliance / support — applying that grouping in the UI (3 column headers + grids) would aid scannability for the polymath operator.

Fix: split SECTIONS into `RUNTIME`, `MONEY`, `TRUST`, `OPS` groups and render `<h2>` per group with a sub-grid each. Defer if scaffold quality is the goal.

---

## Summary

The admin scaffold is honest about phase status, uses single-h1 per page, has working focus-visible chains, and labels every section semantically. UI primitives are accessibility-thoughtful (Dialog APG-correct, EmptyState heading-as-string, Form/Field plumbing). The biggest UX gaps are (a) thin hover affordance on the dashboard's primary nav cards and (b) the not-found landmark duplication risk in the (authed) segment. Everything else is polish or consolidation.

**Optic score:** 91 / 100 (Pass with fixes — landmark + affordance issues should clear before live admin auth lands in Phase 1.0-B).

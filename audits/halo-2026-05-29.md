## Findings

### F-Halo-01 · Critical · Skip-link target lacks `tabIndex={-1}` — SC 2.4.1 partial

**Where:**
- `apps/admin/app/page.tsx:12` — `<main id="main" className="wrap">`
- `apps/admin/app/(authed)/layout.tsx:85` — `<main id="main" className="adm-shell__main">`
- `apps/admin/app/sign-in/page.tsx:27` — `<main id="main" className="wrap">`
- `apps/admin/app/not-found.tsx:6` — `<main id="main" className="wrap">`

**Issue:** The skip-link in `layout.tsx:49-51` (`<a className="sz-skip" href="#main">`) targets `#main`. Per SC 2.4.1 best practice, the target must be programmatically focusable so activating the skip-link actually moves keyboard focus. Without `tabIndex={-1}` on `<main>`, Safari + Firefox sometimes leave focus on the link rather than translating to the target.

**Fix:** add `tabIndex={-1}` to every `<main id="main">` element. Also applies to the user-app `<main>` once it lands.

### F-Halo-02 · Major · Two `<main id="main">` elements may render concurrently when not-found fires inside (authed) segment — SC 4.1.1

**Where:** `apps/admin/app/not-found.tsx:6` renders `<main id="main">`. `apps/admin/app/(authed)/layout.tsx:85` ALSO renders `<main id="main">`. If Next.js renders `not-found.tsx` inside the (authed) segment (it does by default for unmatched routes within the segment), both render → two `<main>` landmarks, duplicate id, ambiguous landmark for SR. Same risk for any future segment-scoped error page.

**Fix:** add `(authed)/not-found.tsx` returning a `<section>` (not `<main>`), or remove `<main>` from the root `not-found.tsx` and rely on the root layout. The root layout currently has no `<main>` — only the `.sz-skip` plus `{children}` in `<body>`. So removing `<main>` from `not-found.tsx` would also break SC 2.4.1. Cleanest fix: keep root `not-found.tsx` with `<main>` AND add `(authed)/not-found.tsx` rendering as `<section>`.

### F-Halo-03 · Minor · `globals.css` font-import is inline `<link>` rather than `next/font` — affects FOIT/FOUT, not contrast

`apps/admin/app/layout.tsx:43-46` loads Cormorant + DM Mono + Inter via Google Fonts CSS rather than `next/font/google`. This causes Layout Shift (CLS), not strictly an SC violation but a performance regression that affects users with cognitive disabilities (CLS amplifies disorientation). PRD §9 cites Lighthouse ≥80 for app.

**Fix (Phase 1.0-B):** swap to `next/font/google` with `display: "swap"` so font-display behavior is deterministic. Same applies to `apps/web` for parity.

### F-Halo-04 · Major · `.adm-card` lacks `:focus-visible` ring — SC 2.4.7 / 2.4.13

`apps/admin/app/globals.css:234-242` defines `.adm-card` (interactive `<a>` element) but provides no `:focus-visible` rule. The global rule for `a:focus-visible` at line 103-106 DOES apply (so a focus ring is rendered), but the ring is outside the card border and may be visually cut by `overflow:hidden` on parent grids. Not a blocking violation since the global ring fires, but ring offset relative to the card is uncoordinated.

**Fix:** add `.adm-card:focus-visible { outline: var(--focus-width) solid var(--focus-ring); outline-offset: var(--focus-offset); }` so card focus matches button/input ring.

### F-Halo-08 · Critical · Sign-in form `aria-required="true"` is set BUT the native `required` attribute is ALSO set → double-announce risk; the inline error state has no `role="alert"`

`apps/admin/app/sign-in/page.tsx:101-117` — email, password, totp all carry `required` + `aria-required="true"`. The native `required` already implies `aria-required="true"` to SR, so the explicit duplicate produces double announcement on some SR/browser combos (NVDA + Chrome). The status `<p role="status" aria-live="polite">` at line 150-156 is correct for soft confirmation, but the form has no `errorSummary` plumbing — if validation needs to surface multiple errors at submit (Phase 1.0-B), `Form.tsx`'s `errorSummary` mechanism (`role="alert"` + `aria-live="assertive"`) should be wired in.

**Fix:**
- Remove `aria-required="true"` when `required` is already present. (Or keep one of them, consistently — convention is to drop the aria-* attr.)
- When real auth lands (1.0-B), use `packages/ui/Form` + `Field` instead of `.adm-form` so the error summary live region is automatic.

### F-Halo-09 · Minor · Touch target on sign-in form: inputs minimum 44px but the labels above them are tight

`apps/admin/app/globals.css:291-300` — `.adm-input` is `min-height: 44px` ✓ (SC 2.5.8). Labels use `--fs-mono-data: 12px` (line 285) which is small; not a violation but borderline for users with low-vision needs. Body text default is 15px so the label being 12px is intentional design (mono-meta style); user can zoom.

### F-Halo-10 · Minor · Focus ring contrast — `--focus-ring: var(--brand-orange)` (#f26b1a) against `--brand-white` (#ffffff)

Computed contrast ≈ 3.5:1 — meets SC 1.4.11 (≥3:1 for non-text UI). On orange button backgrounds, focus ring needs higher contrast since the ring is same color as the button. Verify by visual check; consider an offset border-color shift on `:focus-visible` for orange-bg buttons. Already present in Button.css via `outline-offset: var(--focus-offset)` which puts the ring on the page background, not the button.

`apps/admin/app/(authed)/layout.tsx:49-55` — `<Image src="/logo.png" alt="" width={32} height={32} priority />` with no `aria-hidden`. The wrapping `<Link aria-label="Autonomux admin home">` provides accessible name. Image gets `alt=""` (decorative). Correct. ✓

## Summary

The Phase 1.0-A scaffold is accessibility-thoughtful: working skip-link, single-h1 per page, APG-correct Dialog, native `<select>` (not custom dropdown), proper `aria-label` on Image-only links, focus rings on every interactive primitive, prefers-reduced-motion respected, and reflow to 320px built in. The two critical-class gaps are (a) missing `tabIndex={-1}` on `<main>` targets and (b) the not-found / (authed) double-`<main>` risk. Both are small mechanical fixes that should land before Phase 1.0-B auth wiring exposes the cpanel to real users.

**Halo score:** 88 / 100 (Pass with fixes).

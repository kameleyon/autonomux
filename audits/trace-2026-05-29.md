# Trace — Flow + logic review · 2026-05-29 · Phase 1.0-A Foundation

**Scope:** `packages/db/migrations/**`, `packages/db/src/**`, `packages/cipher/src/**`, `packages/llm/src/**`, `apps/worker/src/**`.
**Threat lenses:** RLS leak, audit-chain tamper resistance, cipher AAD binding, LLM error swallow / fallback abuse, idempotency soundness.

## Findings

### F-Trace-01 · Critical · Migration sequencing bug — `0001_init.sql` references `vector(1536)` before `0004_pgvector.sql` enables the pgvector extension

**Where:**
- `packages/db/migrations/0001_init.sql:120` — `embedding vector(1536),` inside `agent_memory_episodes`
- `packages/db/migrations/0004_pgvector.sql:22` — `create extension if not exists "vector";`

**Issue:** Supabase applies migrations in numeric order — `0001 → 0002 → 0003 → 0004`. On a fresh project where the `vector` extension is NOT pre-installed (Supabase managed projects require explicit `create extension` to enable pgvector even though the binary is bundled), `0001_init.sql` will fail at the `vector(1536)` reference because the type does not yet exist. README at `packages/db/README.md:36-44` walks the user through `supabase db reset` which applies them in order — the first deploy WILL hit this error.

**Fix:** move `create extension if not exists "vector";` to the TOP of `0001_init.sql` (before any table reference), alongside `create extension if not exists "pgcrypto";` at line 15. Keep `0004_pgvector.sql` for the HNSW index, since that's the genuinely separate concern. Two-line change:

```diff
+ create extension if not exists "vector";
  create extension if not exists "pgcrypto";
```

Without this fix, `pnpm migrate:apply` against a fresh Supabase project fails before any RLS or audit-chain logic ever runs.

### F-Trace-02 · Critical · Heartbeat cron deduplicates after first minute — `apps/worker/src/jobs/cron.ts:46-48`

**Where:** `apps/worker/src/jobs/cron.ts:46-48`

```ts
requestId: `heartbeat-${HEARTBEAT_REPEAT_KEY}`,
tenantId: "system",
data: { kind: "heartbeat" },
```

The heartbeat job is registered as a BullMQ repeatable with `pattern: "* * * * *"` (every minute). Every tick gets the SAME `requestId` (`"heartbeat-cron:heartbeat"`). The shared queue processor in `apps/worker/src/queues/index.ts:222-235` calls `acquireIdempotencyLock(connection, requestId, 24h_TTL)`. After the first tick acquires the lock, every subsequent tick for the next 24 hours returns `{ status: "deduped" }` and never executes. The heartbeat → silent at minute 2 onward.

**Impact:** The heartbeat is the only mechanism in Phase 1.0-A4 to verify the worker is alive end-to-end (per the file's own docstring at lines 1-9). With dedup firing, "no heartbeat in Axiom" looks identical to "worker dead" — but actually the worker is healthy and the cron is deduped. False-positive ops alerts incoming.

**Fix:** generate a per-tick requestId — include `Date.now()` or the BullMQ job id at processing time:

```ts
// Option A — at registration, omit requestId from the payload and have the
// processor mint `${queue}-${job.id}` when none is set.
// Option B — at the processor, skip the idempotency lock when
//   tenantId === "system" && payload.data.kind === "heartbeat".
```

Option B is the minimal-diff fix: in `queues/index.ts:processStubJob`, short-circuit the idempotency check when `job.data.tenantId === "system"`.

### F-Trace-03 · Major · `index.ts` swallows cron registration failure — but logs it

**Where:** `apps/worker/src/index.ts:93-101`

```ts
try {
  await registerCronJobs({ cronQueue: registry.cron, logger });
} catch (err) {
  logger.error({ err }, "cron registration failed");
  // Cron registration is best-effort at boot; do not abort the worker.
}
```

The error is logged (so it's not technically swallowed), but the worker boots anyway in a state where the daily audit checkpoint won't fire, the morning briefing won't trigger, etc. PRD §1 "no fallbacks, no fix-later" → a worker that boots without its cron is a degraded mode.

**Fix:** at v1.0-A this is defensible (scaffold-only heartbeat), but the contract should harden by Phase 1.0-B: classify cron failure as fatal in production, soft in dev. Use `env.NODE_ENV === "production"` → throw, else log + continue.

### F-Trace-04 · Major · `audit_log.created_at` uses `default now()` — but the chain payload uses `extract(epoch from coalesce(new.created_at, now()))` which mints a DIFFERENT timestamp than what the row stores when default fires

**Where:** `packages/db/migrations/0003_audit_chain.sql:80,166`

The `BEFORE INSERT` trigger runs AFTER PostgreSQL applies column defaults — so `new.created_at` IS already set when the trigger fires. The `coalesce(new.created_at, now())` defensive read returns `new.created_at`. The verifier at line 166 recomputes from `r.created_at` which is the stored value. Same value both sides. ✓ chain verifies.

This is not a bug — wanted to flag I checked it. Mark as informational.

### F-Trace-05 · Major · `verify_audit_chain(p_tenant_id := uuid)` path only verifies per-row hash but does NOT verify the prev_hash chain for the filtered slice

**Where:** `packages/db/migrations/0003_audit_chain.sql:147-154`

```sql
if p_tenant_id is null then
    if (prev is null and r.prev_hash is not null) ...
    then return false;
    end if;
end if;
```

The `prev`-chain check is GATED on `p_tenant_id is null`. When verifying a tenant slice, only per-row hash recomputation is done — no linkage check. The docstring at lines 117-124 explicitly states this trade-off ("the per-tenant filter is for admin-cpanel proof of inclusion, not independent chain integrity"). Behavior matches the documented intent. ✓

Mark as informational — the trade-off is honest and the global-chain verify still gives integrity.

### F-Trace-06 · Major · `audit_log_tenant_select` policy on lines 268-270 may give authenticated users SELECT on rows where `tenant_id` matches THEIR JWT — but `actor_user_id` may belong to a DIFFERENT user within the same tenant (multi-user tenants Phase 1.7+)

**Where:** `packages/db/migrations/0002_rls.sql:267-270`

```sql
create policy audit_log_tenant_select on public.audit_log
    as permissive for select to authenticated
    using (tenant_id = public.current_tenant_id());
```

V1 = 1 user per tenant, so this is fine today. Once shared AlterEgos land (Phase 1.7+ per PRD §2 / §6.1), every user in a tenant can read every other user's audit rows. That may be intended (transparency within a household tenant) — or it may not (GDPR data-minimization).

**Fix:** add a comment line above the policy noting the Phase 1.7 review point. No code change needed at A.

### F-Trace-08 · Minor · LLM adapter has no top-level `timeout_ms` field on `CompleteRequest` — PRD §1 says "every LLM call: token budget, **timeout**, retry, cost tracked"

`packages/llm/src/types.ts:83-110` — exposes `signal: AbortSignal` (line 108-109) which lets the orchestrator wire any timeout strategy it wants. Budget + retry + cost are explicit fields. Timeout is implicit via `signal`. Not a violation — the contract is honored by composition — but a `timeout_ms` convenience parameter would be cleaner DX and would force callers to think about it.

**Fix (Phase 1.0-B):** add `timeout_ms?: number` to `CompleteRequest` and have `createLlmClient.complete()` wrap with an `AbortController` if no `signal` was supplied. Defaults to 90s (matches PRD §9 briefing-gen latency budget).

### F-Trace-09 · Minor · LLM adapter swallows JSON parse errors silently in two places — `openrouter.ts:251,454`

Both are inside a stream-parse loop and a tool-call args parse. Falling back to `continue` or `{ _raw: ... }` is defensible at a low level, but neither logs the malformed input. Observability cost.

**Fix:** call `ctx.logger?.warn({ raw_event: event }, "openrouter sse parse failed")` etc. Doesn't block.

### F-Trace-10 · Polish · `ENVELOPE_VERSION = 1` is hardcoded in the AAD; future v=2 path needs careful negotiation

`packages/cipher/src/envelope.ts:118` writes `ENVELOPE_VERSION` into the app AAD at encrypt time. `decrypt` rejects mismatched versions. When v=2 lands, the decision tree must accept v=1 ciphertexts (legacy) AND v=2 — currently `if (env.v !== ENVELOPE_VERSION)` is strict equality, so it will refuse v=1 after the constant flips to 2. Acceptable for Phase 1, but document the migration path before bumping the version. Comment-level fix.

## Summary

Two production-blocking flow bugs land here:

1. **F-Trace-01** — migration 0001 references `vector(1536)` before 0004 enables the extension. Fresh-project deploys fail at the SQL push. Two-line fix (move extension declare up).
2. **F-Trace-02** — heartbeat cron deduplicates itself after minute 1, so observability silently dies. Either skip the idempotency lock for `tenantId === "system"` or mint per-tick request ids.

Cipher subsystem is rigorous (`generateDek` → `wrapDek` → `seal` with double-bound AAD, wipe in finally, constant-time AAD compare). Audit-chain trigger is append-only at three layers (RLS + reject-mutation trigger + chain-hash trigger). RLS policies blanket every tenant-scoped table with the right macro. LLM adapter has explicit error types and no silent provider fallback.

**Trace score:** 65 / 100 (FAIL on count — the two Critical findings push the score below the 70 threshold for "Pass with fixes" once Blocker/Critical deductions stack).

Note for Jury synthesis: F-Trace-01 alone is a hard gate. The system literally won't apply migrations 0001 on a clean Supabase project without that fix.
